The Traffic simulation lets the Operations Manager rehearse the same pressure from the fleet side: how vehicles, chargers, and the road network hold up when demand rises. The shift it gives the operator: from finding out something is wrong too late, to always knowing what is running, what is at risk, and what is out of service. In the Operations Manager view, scoped to Turtle Bay here, Simulation sits alongside Fleet, Utilization, Drivers, and EV.

Every run follows the same arc: you set the operating conditions, the model forecasts utilization and charging strain, and it proposes the repositioning and pooling moves that recover headroom. Where Crowd is read from the guest's side, waits and station demand, Traffic is read from the asset's side, utilization, idle time, charging, and driver fatigue. Same day, same island, the operational half of the picture.

## Before running

![Traffic simulation before a run: the left panel sets the forecast type, scenario, fleet assumptions, buildings, and vehicle mix, with the current readings on the right.](08-simulation-traffic-before.jpg)

Before running sets the operating conditions for the day. The Scenario preset is the headline assumption, Maintenance for instance takes assets offline and tightens everything downstream. The fleet assumptions follow, and Speed on road matters more than it looks: it sets how fast a vehicle completes a leg and returns to service. Lower the fleet or the chargers, or drop road speed, and the same guest volume produces a sharper utilization and charging forecast, which is how you find the point where the day stops being comfortable.

| Control | What it sets |
| --- | --- |
| Forecast type | Vehicle, Charging, or Combined |
| Scenario preset | Normal, Peak, Surge, or Maintenance |
| Guest volume | Expected on-island, here 2,500 |
| Number of vehicles | Available on-island, here 800 |
| Vehicle capacity | Average per vehicle, here 2,500 |
| Active EV chargers | The active charger count |
| Speed on road | Average km per hour, here 100 |
| Buildings | Marina Hub, Spine Hub, Bridge Point, Beach Cluster, toggled in or out of scope |
| Vehicle types | Shuttles (16, 92% util), Buggies (22, 78% util), Tenders (6, marina), VIP cars (2, reserved) |

The right panel holds the live baseline: transfer variance +18%, 412 booking requests, 28 of 46 vehicles free, average request age 2.4 minutes, top station requesting Marina.

## Reading the fleet forecast

Read the verdict first, then the metric that drove it: utilization and charging set capacity risk, fatigue and maintenance set continuity risk.

| Metric | What it measures | Watch when |
| --- | --- | --- |
| Utilization at peak | Share of the fleet in use at peak | Approaches 100%, here 96% |
| Idle time | Share of the fleet sitting unused | Very low means over-stretched, very high means waste |
| Charging demand | Load on the EV chargers | Chargers hit capacity, here 3 maxed at 84% |
| Energy needed | Extra energy the scenario draws | Rises against typical, here +620 kWh |
| Maintenance backlog | Vehicles waiting on service | Grows across the shift, here +2 by end of shift |
| Fatigue risk | Drivers near a fatigue limit | Any count above zero, here 3 on the bridge |

Three rules sharpen the read:

- Utilization and idle time go together. 96% utilization with 6% idle means no buffer; high utilization with healthy idle is a busy day, high utilization with near-zero idle is a fragile one.
- Treat charging as its own ceiling. Vehicles needed can look solvable while charging at 84% with three chargers maxed is the real limit, you cannot dispatch what cannot charge.
- Read the peak, not the average. The metrics are framed at the peak because that is where the day fails; a comfortable daily average can still hide a peak that tips the fleet over.

## Implementing a plan

Once a run is forecasted, the right panel turns the result into action. It lists the plans the operations persona prepared for the event, each with its forecasted effect and an Implement control. Selecting Implement enacts that plan's actions: the dispatch goes to the fleet and building teams, the action moves into the live operations view, and the forecast baseline re-anchors to track the plan as it runs. This is the step that takes a run from a what-if to an executed response.

A plan can be implemented as forecasted, or adjusted first by reopening it under Action plan, editing its actions, and re-running to confirm the new effect before dispatch.

## Creating an event

Under Event to simulate, New Event opens a builder for a condition the presets do not cover. You name the event and set its effect, then it joins the event list and can be selected into any run alongside the built-in events. This lets the operations persona model a specific operational reality, a maintenance window, known road works, a depleted depot, rather than only Normal, Peak, Surge, and Maintenance.

## Creating an action plan

Under Action plan, View All lists the saved plans and New Plan creates one. A plan is a named set of actions, with the action count shown beside it. Once saved, a custom plan is selectable and forecasts against any event the same way as the built-in plans, so the operations persona can prepare a standard response to a recurring strain, the peak-hour pooling play, the maintenance-day reroute, and confirm its forecasted effect before running it live.
