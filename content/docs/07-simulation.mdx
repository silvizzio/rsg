Simulation lets operators rehearse a what-if on the twin without touching live data. You build a scenario, run it, and read the forecast the system returns, with the same recommended-action thinking that appears in live operation. It is a way to walk a busy day through from start to finish before it ever happens.

Simulation opens from the top button on the right toolbar.

## Before running: building a scenario

This is the setup screen before a run, shown here in the Guest Experience view with the Pre-Trip phase selected.

![The simulation setup before running. The left rail builds the scenario, the right rail shows the live baseline it runs against, and Run Simulation starts it.](07-simulation-before.jpg)

The left rail is where you build the scenario, read top to bottom. Event to simulate is the list of conditions to model, such as a Peak Arrival Surge or a VIP Convoy, each with a one-line note on its effect; tick the ones you want, or use New Event to add your own. Guest volume sets how many guests are expected on the island. Intensity preset, Normal, Peak, Surge, or VIP, is a one-tap starting point that sets the scenario to a known pattern you can then fine-tune. Pre-trip variables hold the assumptions the run leans on, here the marina capacity per hour and the buffer time. Response plan is the response you want to test against the event, from Baseline, which does nothing, through Recall and Relieve or Reroute and Advise, each showing how many actions it would take. The right rail shows the live baseline the run measures against: the analysis window, transfer variance, booking requests, available vehicles, average request age, and the stations asking for the most transfers. When the scenario is set, Run Simulation produces the forecast.

## Reading the forecast

<div className="doc-split-text">

The forecast panel is the result of the run, read top to bottom. It opens with the forecasted outcome, a plain risk verdict such as Moderate Risk, with a one-line summary of what the run predicts and whether action is needed to hold the service level. Below it, projected key metrics give the headline numbers the run expects, each shown against now so you can see the change: guests, wait time, transfer variance, available vehicles, average request age, crowd risk zones, capacity gap, and affected stations. At the bottom, the forecast spec records how the run was made so it can be trusted and repeated: the model, the forecast type, the window, the buildings and stations included, the scenario, the seed, and the time of the last run.

## Crowd and traffic on the map

While a simulation runs, crowd density and traffic flow are drawn straight onto the map, so you can see where people and vehicles build up as the scenario plays. Density reads as High, Medium, or Low, and a sustained High area is what raises a crowd density alert in live operation.

## Running a simulation

1. Open simulation from the top button on the right toolbar.
2. Pick the events to simulate and choose an intensity preset.
3. Set the guest volume and the pre-trip variables.
4. Choose the response plan you want to test.
5. Run the simulation and read the forecast.
6. Close simulation to return to live data.

<Callout type="warning">
While simulation is on, the alert tray shows scenario alerts, not live ones. Live incidents are held until you close simulation.
</Callout>
